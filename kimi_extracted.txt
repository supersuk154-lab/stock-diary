1. 아키텍처 & 모듈화 ⭐⭐⭐⭐☆
잘된 점
app.py를 진입점으로 하고, 기능별로 tab_*.py, auth.py, db.py, prices.py 등으로 깔끔하게 분리했습니다.
관심사 분리가 명확: UI 탭 / DB 쿼리 / 주가 조회 / 인증 / AI 호출이 각자 파일을 가짐.
ui_components.py로 card, banner, render_radar_chart 등을 재사용 가능하게 추출한 점이 좋습니다.
아쉬운 점
tab_diary.py가 너무 비대합니다 (500+ 라인). 투자 능력치, 배당금, 멘토 채팅, 이미지 업로드, 매매 확인, 최종 분석 등 여러 기능이 한 파일에 몰려 있습니다.→ diary_*.py (예: diary_chat.py, diary_inventory.py, diary_upload.py) 로 추가 분해 권장.
app_old.py가 여전히 UTF-16LE 깨진 상태로 남아있는데, 버전 관리(Git)에 맡기고 삭제하는 편이 낫습니다.

2. 코드 품질 & 가독성 ⭐⭐⭐☆☆
잘된 점
한국어 주석과 문자열이 많음에도 코드 흐름이 직관적입니다.
safe_generate() 같은 래퍼로 AI API 호출을 추상화한 것은 좋은 설계입니다.
to_kst_str() 등 유틸리티 함수가 명확합니다.
아쉬운 점
매직 넘버/문자열 남발:KR_MIN_WAGE_2026 = 10_320은 상수화했지만, "#월급날정기매수", "#배당금달달해" 같은 태그 문자열이 db.py, tab_diary.py 등 여기저기 하드코딩되어 있습니다.→ constants.py에 TAG_ROUTINE_SALARY = "💸 #월급날정기매수"처럼 중앙 집중화하면 유지보수가 쉬워집니다.
함수 길이: render_diary_tab()은 사실상 하나의 거대한 스크립트처럼 동작합니다. 내부 로직을 더 작은 함수로 쪼개면 테스트와 가독성이 크게 향상됩니다.
주석 품질: 일부 주석이 코드와 동기화되지 않은 느낌입니다. 예를 들어 tab_diary.py 상단에 get_dividend_work_stats 함수가 정의되어 있는데, 이 함수는 db.py에 있는 것이 더 자연스럽습니다.

3. Streamlit 사용법 ⭐⭐⭐⭐☆
잘된 점
st.session_state를 적극 활용하여 멀티스텝 폼(upload → verify → ask_next → final_analysis)을 구현한 것은 Streamlit에서 꽤 정교한 패턴입니다.
st.cache_data를 DB 조회·주가 조회에 잘 적용했습니다.
st.chat_message + st.form을 조합해 챗봇 UI를 만든 것이 창의적입니다.
토스 스타일 CSS를 주입해 기본 Streamlit UI를 커스터마이징한 노력이 보입니다.
아쉬운 점
st.rerun() 호출이 너무 잦습니다. 태그 선택, 멘토 선택, 채팅 입력 등 거의 모든 인터랙션에서 rerun이 발생합니다.→ st.session_state 플래그를 더 세밀하게 관리하면 불필요한 전체 재실행을 줄일 수 있습니다.
key=f"uploader_{st.session_state['uploader_key']}" 같은 패턴은 Streamlit의 위젯 키 충돌을 피하기 위한 꼼수인데, 이해하기 어렵고 버그 유발 가능성이 있습니다. 주석으로意圖를 더 명확히 적어주면 좋겠습니다.

4. 데이터베이스 & 보안 ⭐⭐⭐⭐☆
잘된 점
Supabase를 사용하고, RLS(Row Level Security)를 언급하며 보안 의식이 있습니다.
user_id 기준으로 데이터를 필터링하여 타인 데이터 노출 방지.
비밀번호 변경, 계정 삭제(복구 불가) 등 안전한 플로우를 제공합니다.
배당금 데이터의 구/신 스키마 호환(dividend_amount vs quantity)을 고려한 점이 세심합니다.
아쉬운 점
tab_settings.py의 계정 삭제 후 st.rerun()만 호출하는데, 실제로는 로그아웃 처리(세션 클리어)가 함께 필요합니다. 현재 코드에서는 삭제 후에도 세션이 남아있을 수 있습니다.
get_real_inventory()에서 user_id를 파라미터로 받지만, 쿼리에 .eq("user_id", user_id) 조건이 누락된 것처럼 보입니다. 현재는 .neq("type", "dividend").execute()만 하는데, 이는 모든 사용자의 매매 데이터를 긁어옵니다.→ 심각한 보안 버그 가능성. db.py의 get_real_inventory와 get_dividend_total 모두 user_id 필터링이 필요합니다.

5. AI 통합 (Gemini) ⭐⭐⭐⭐☆
잘된 점
google-genai SDK를 사용하여 최신 방식으로 Gemini를 호출합니다.
types.GenerateContentConfig + system_instruction + response_mime_type="application/json"을 사용해 Structured Output을 시도한 것은 훌륭합니다.
멘토 페르소나(심리상담가/찐친/1타강사)별로 프롬프트를 동적으로 구성하는 로직이 재미있고 실용적입니다.
이미지(MTS 캡처)에서 JSON으로 종목·수량을 추출하는 파이프라인이 잘 설계되어 있습니다.
아쉬운 점
safe_generate의 fallback 처리는 좋지만, AI 응답이 JSON이 아닌 경우(예: 안전 필터, 잘못된 포맷)의 복구 로직이 부족합니다. json.JSONDecodeError를 잡아내지만, 사용자 경험 측면에서 "다시 시도" 외에 별다른 수단이 없습니다.
MODEL_NAME = "gemini-3.1-flash-lite-preview" — 프리뷰 모델은 언제든 변경/삭제될 수 있어서, 프로덕션에서는 안정 버전(gemini-1.5-flash 등)을 쓰는 게 더 안전합니다.

6. 에러 처리 ⭐⭐⭐☆☆
잘된 점
대부분의 DB/API 호출을 try/except로 감싸고 있습니다.
safe_generate처럼 AI 호출은 중앙에서 fallback 메시지를 반환합니다.
아쉬운 점
except Exception:으로 넓게 잡고, 구체적인 예외 타입을 구분하지 않는 경우가 많습니다.→ supabase.PostgrestAPIError, yf.YFinanceException 등을 구분하면 디버깅과 사용자 메시지가 훨씬 정교해집니다.
st.error()로 에러를 표시한 후 st.stop()이나 return 없이 코드가 계속 진행되는 부분이 있습니다. 예: tab_diary.py의 get_real_inventory 실패 후에도 아래 UI가 그려지려 할 수 있습니다.

7. UI/UX ⭐⭐⭐⭐☆
잘된 점
동굴 모드(Zen Mode): 시장 폭락 시 숫자를 가리고 멘탈을 다잡는 기능 — 사용자의 심리적 상태를 고려한 훌륭한 기획입니다.
오늘의 알바생: 배당금을 최저시급으로 환산해 "내 주식이 나 대신 몇 시간 일했는지" 보여주는 것, 기획력이 돋보입니다.
가족 분담금: ETF는 '막내', 대장주는 '맏형'으로 은유하는 것이 재미있고 직관적입니다.
FIRE 카운트다운: 월 배당 목표를 시각화한 점도 동기부여에 좋습니다.
아쉬운 점
CSS 주입(toss_style)이 app.py에 하드코딩되어 있고, !important 남발. Streamlit 버전이 올라가면 깨질 위험이 큽니다.→ st.html() (Streamlit 1.37+)나 커스텀 컴포넌트로 분리하는 것을 고려하세요.
tab_diary.py의 채팅 인터페이스가 st.form + st.text_area 조합인데, st.chat_input을 쓰면 더 자연스럽습니다. (이미 시도했지만 "페이지 맨 아래에 고정되어 안 보일 수 있다"는 이유로 바꾼 것 같습니다. 이는 Streamlit 버전에 따라 다릅니다.)
